## 1. V2 不做什么

以下边界永久锁定，V2/V3/V4 均不突破：

- 不做全自动采购
- 不做自动发布商品
- 不做自动投广告
- 不做外部平台自动操作（登录、下单、联系供应商、改库存/价格、处理订单/退款）
- 不承诺 AI 结果一定正确
- 不在用户未确认的情况下对外执行任何资金/账号/客户操作

---

## 2. V2 MVP 做什么

从"多个独立分析页面各自跑"升级为"一条工作流串起来"：

### 2.1 入口
- 用户在 `/opportunities` 机会雷达中看到候选品排行榜
- 选择 1 个候选商品 → 进入 V2 工作流（`/v2-lab` 或 `/workflow`）

### 2.2 工作流步骤（Phase 1A 收敛为 4 步）

| 步骤 | 名称 | 说明 | 复用现有能力 |
|------|------|------|-------------|
| 1 | 货源确认 | 1688 搜索词、MOQ、价格带、供应商数量、采购难度 | `/api/agents/sourcing` |
| 2 | 风险排查 | 认证、侵权、物流、售后、平台规则 | `/api/agents/risk` |
| 3 | 利润与合规粗判 | 售价、成本、运费、佣金、毛利率 + 目标市场认证要求 | `/products/new` 利润 + `/api/agents/summary`（合规部分） |
| 4 | 小白结论与下一步 | 综合判断：继续验证 / 小单测试 / 放弃 + 人工下一步动作 | `/api/agents/summary`（新手结论） |

### 2.3 人工确认机制
- 每个步骤卡片内嵌「人工确认」checkbox
- 4 步全部确认后，底部按钮变为「保存到任务中心」
- 保存到 `/tasks`，类型 `workflow`，附带 decisionStatus

### 2.4 不自动执行
- 所有「下一步动作」只是建议，不是自动化操作
- 不自动联系供应商 / 下单 / 上架 / 对外发送任何内容

## 4. V2 成功标准

- [ ] 用户能看懂下一步该干嘛
- [ ] 用户知道哪些步骤需要人工确认
- [ ] 用户不会误以为系统能自动赚钱
- [ ] 系统不扩大真实 AI 成本（V2 初期 0 AI 调用）
- [ ] 不影响 V1.5 生产稳定版
- [ ] `/v2-lab` 不与现有页面冲突
- [ ] mock 原型获得用户认可后再进入 Phase 1B

## 6. 实施阶段（细化版）

### Phase 0 ✅ 已完成 — 沙盒规划 + 初始 mock
- 规划文档
- `/v2-lab` 初始静态 mock（三栏，6 步，独立 check 右栏）
- 不接 API、不接 AI、不写 DB

### Phase 1A ✅ 已完成 — 交互原型收敛
- 三栏改两栏：去掉独立右栏 checklist
- 6 步合并为 4 步
- checkbox 内嵌到每个步骤卡片
- 每个步骤卡片含风险提醒
- 顶部文案明确"不调 AI、不自动操作"
- 不接 API、不接 AI、不写 DB

### Phase 1B ✅ 已完成 — 只读已有 opportunities 任务记录
- 新增 `components/cross-border/V2WorkflowLabClient.tsx`（客户端组件）
- 新增 `lib/agents/v2WorkflowRecords.ts`：纯函数 `extractCandidates()`，解析 opportunities 记录
- 新增 `lib/agents/v2WorkflowRecords.test.ts`：13 个测试，覆盖正常/缺失/异常字段
- 使用 `useAccessPassword()` + `GET /api/tasks?type=opportunities&limit=1` 读取最新记录
- 左侧候选品列表来自真实 tasks 数据，替代硬编码 mock
- 右侧 4 步工作流根据选中候选品的已有 sourcing/risk/summary 摘要填充
- 处理了 6 种状态：loading / no_password / password_expired / unauthorized / error / empty
- 不调用新 AI，不保存任务，不新增 API
- checkbox 仍为本地演示状态

### Phase 1C — 单商品串行调用（用户确认后）
- 用户在 Phase 1B 基础上点击「重新分析」
- 串行调用 `/api/agents/sourcing` → `/api/agents/risk` → `/api/agents/summary`
- 最多 3 次 AI 调用 / 商品，不批量，不自动
- 需访问密码保护

### Phase 1D — 保存 workflow 任务
- 4 步全部人工确认后，POST `/api/tasks`（type=`workflow`）
- 不新增数据库表，复用现有 `ViralAnalysisRecord`
- `/tasks` 页面可筛选 `workflow` 类型

### Phase 2 — 工作流增强（待定）
- 从 `/opportunities` 直接跳转 `/v2-lab?product=xxx`
- 支持跳过/重试单步
- 利润粗算模块独立

### Phase 3 — 正式发布（待定）
- 页面改名 `/workflow`
- 加入主导航
- 生产部署

## 7. 设计决策记录

| 决策 | 原因 |
|------|------|
| 三栏改两栏 | 右栏 checklist 与中栏步骤重复，认知负担高 |
| 6 步合并为 4 步 | 「合规确认」和「新手适合度」都来自 summary；「下一步动作」并入结论 |
| checkbox 进卡片 | 减少视线跳动，确认动作与步骤摘要在同一上下文 |
| Phase 1B 不调新 AI | 先验证交互流程，再渐进接入 AI；避免 mock → 大规模 AI 跳跃 |
| 不新增 API | Phase 1C 之前全部复用现有 Agent API |

---

## 8. 风险与约束

- V2 不能导致 AI 调用次数激增（Phase 1C 单商品最多 3 次）
- 工作流步骤顺序需保持灵活性（可跳过、可重试）
- mock 原型不能给用户造成"功能已可用"的错觉
- 沙盒分支不 merge 前需经过生产 review
