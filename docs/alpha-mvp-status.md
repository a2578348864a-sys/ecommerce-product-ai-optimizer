# Alpha MVP 状态说明

> 最后更新：2026-06-24 · Phase 2-O · 生产 HEAD `43349a6`

## 1. 项目定位

**轻选 Agent** 是跨境电商运营全流程 Agent 工作台。AI 给建议、生成资料、提示风险，关键动作人工确认，未来逐步走向受控自动化。

当前处于 **Alpha MVP / Pre-commercial** 阶段，聚焦前半段运营链路：市场机会发现 → 选品立项 → 任务沉淀 / 运营复盘。

## 2. 当前已完成主链路

```
首页工作台（/）
  → 机会候选池（/opportunities）
    → 单品分析（/workflow）
      → 保存到任务中心
        → 任务中心运营跟进台（/tasks）
```

### 2.1 首页工作台

- 路由：`/`
- 首屏展示候选池概况、任务跟进统计、最近分析和推荐下一步
- 空状态显示"新手三步开始"
- 工作流路径卡片：找机会 / 单品分析 / 任务中心 / 批量分析
- 数据来源：浏览器 localStorage + 任务 API（只读）

### 2.2 机会候选池

- 路由：`/opportunities`
- 标题：机会雷达 / 候选品池
- 支持：手动输入候选商品 → AI 分析 → 筛选/排序/状态标记
- 候选品保存在浏览器 localStorage，7 天 TTL
- 支持 5 种状态：待判断、值得深挖、已进入单品分析、暂缓、放弃
- 支持轻量公开线索抓取（遵守 robots.txt，不调用 AI，最多 5 个 URL）
- 链接到单品分析：点击"用单品分析深挖"跳转 `/workflow`

### 2.3 单品分析

- 路由：`/workflow`
- 输入一个商品 → 自动完成 4 步 AI 分析：
  1. 货源判断
  2. 风险排查
  3. 小白结论
  4. 上架文案
- 分析结果 2 小时本地持久化（localStorage）
- 支持人工逐项复核（StepReviewCard × 4）
- 保存到任务中心（调用 save-task API）
- 支持从机会雷达带入商品名和来源信息

### 2.4 批量分析

- 路由：`/workflow/batch`
- 最多 3 个商品快速对比
- 支持 CSV/TXT 文件导入
- 前端串行执行，分析完成后可保存结果
- 结果本地持久化，关闭/刷新可恢复

### 2.5 任务中心运营跟进台

- 路由：`/tasks`（列表）、`/tasks/[id]`（详情/运营跟进面板）
- 任务列表显示运营摘要：优先级、风险、新手适配、下一步
- 支持搜索、类型筛选、人工决策状态筛选、Agent 状态前端筛选
- 任务详情首屏为"运营跟进面板"：AI 辅助判断、建议动作、人工决策下拉
- 人工决策状态：待判断、可继续、需补资料、已淘汰
- 支持查看和更新复核状态

### 2.6 辅助页面

| 路由 | 说明 |
|------|------|
| `/agent` | 能力路线图与阶段说明，不是当前主流程入口 |
| `/agent/run` | Agent 主链路驾驶舱 / 实验看板 |
| `/viral` | 海外爆款趋势拆解（模拟/AI 辅助） |
| `/sourcing` `/risk` `/summary` `/products/new` `/materials` | 旧版独立分析页，保留但主入口已收敛 |

## 3. 当前不是

- ❌ 不是成熟全自动商业系统
- ❌ 不是自动采购、自动上架、自动投放工具
- ❌ 不是跨设备云端候选池（候选池基于浏览器 localStorage）
- ❌ 不是无人值守执行台
- ❌ 当前不会自动操作真实平台账号

## 4. 当前技术边界

| 组件 | 实现 | 限制 |
|------|------|------|
| 候选池 | 浏览器 localStorage | 本设备 7 天，不跨设备 |
| 单品分析缓存 | 浏览器 localStorage | 2 小时 TTL，不跨设备 |
| 真实 AI 调用 | DeepSeek chat 通过 OpenAI SDK | 需要访问密码 |
| 任务沉淀 | Prisma + SQLite | 单用户，无权限分层 |
| 生产部署 | PM2 + Next.js on 阿里云 | 127.0.0.1:3005 + Nginx 反代 |
| 访问控制 | 环境变量密码 + 前端 localStorage TTL | 单密码，无多用户 |

## 5. 当前可演示路径

1. 打开首页 `/`
   → 看到候选池概况、任务跟进统计、推荐下一步
2. 进入机会候选池 `/opportunities`
   → 认识候选品池的概念：先筛选标记，再深挖
3. 选择一个候选品 → 点击"用单品分析深挖"
   → 进入 `/workflow`，商品名和来源已带入
4. 点击"开始一键分析"
   → AI 依次完成 4 步分析（货源、风险、结论、文案）
5. 逐项人工复核 → 保存到任务中心
6. 进入任务中心 `/tasks`
   → 查看运营摘要（优先级、风险、下一步）
7. 打开任务详情 → 运营跟进面板
   → 查看 AI 判断、建议动作、更新人工决策状态

## 6. 当前剩余风险

| 风险 | 说明 |
|------|------|
| 候选池 localStorage MVP | 浏览器清除数据则候选池丢失，不跨设备同步 |
| 真实机会来源偏手动 | 公开线索抓取最多 5 个 URL，需手动输入候选品 |
| 自动爬虫未做 | 大规模自动发现候选品的能力尚未开发 |
| 服务端候选池未做 | 候选品无法跨用户共享或持久化到服务端 |
| 单用户 | 无多用户隔离、权限、配额 |
| AI 成本未受控 | 无 token 预算、无调用次数限制 |
| 旧版独立分析页 | `/sourcing` `/risk` `/summary` 等页面仍存在但非主链路入口 |

## 7. 下一阶段建议

### Option A: Phase 3-A — 服务端候选池设计评估
- 评估将 localStorage 候选池迁移到服务端的方案
- 不改 DB schema，先做只读方案设计
- 目标：候选池可跨设备、可持久化

### Option B: Phase 3-A — 真实数据源管理方案
- 评估合规的公开数据源（RSS、Sitemap、官方 API）
- 只读方案设计，不改现有抓取逻辑
- 目标：减少手动输入候选品的依赖

> 以上仅为建议，不包含在本轮交付包内。具体方向由用户确认后进入下一阶段。
